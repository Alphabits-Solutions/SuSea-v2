Your AI agent aced every test case. The demo went flawlessly. Then you deployed it to real users — and within days, it was looping, hallucinating, or silently producing wrong answers that nobody caught until a client complained.

This is not a fringe scenario. It is the normal trajectory for AI agents that were validated in controlled conditions but never stress-tested against the chaos of production. Here are the five failure modes we encounter most often, what causes them, and how to fix them.

---

## 1. Prompt Drift

**What it is:** Your system prompt was written to handle the workflows you anticipated at launch. Production is full of workflows you did not anticipate.

As your product evolves — new features, new integrations, new edge cases — the gap between what the prompt was written for and what the agent is actually being asked to do widens. The agent's behaviour degrades gradually, not dramatically, which makes it easy to miss.

A customer support agent trained on v1 of your product documentation starts confidently answering questions about v3 features it has never been told about. A sales agent given a new CRM integration continues routing leads through the old workflow because the prompt was never updated.

**How to fix it:** Treat your system prompt as a versioned artefact. Every time a tool is added, a workflow changes, or a failure rate spikes, audit the prompt against current reality. Add a regression suite of 20–30 representative inputs that runs automatically on every prompt change.

---

## 2. Tool Misuse Under Edge Cases

**What it is:** Agents use tools correctly on the happy path. On edge cases, they call tools with malformed inputs, call them in the wrong order, or call them when the preconditions for a valid call are not met.

A booking agent that works perfectly for standard appointments will call the calendar API with a null start time when the user provides an ambiguous date like "sometime next week." The API returns a 400 error. The agent has no fallback — so it either loops trying to fix it or halts entirely.

This is rarely a problem in testing because testers provide well-formed inputs. Real users do not.

**How to fix it:** Add input validation at the tool-call layer, not just inside the tool itself. Define explicit preconditions for each tool in the agent's instructions, and add a fallback branch for every tool that can return a non-200 response. Log every tool invocation with its full input payload so you can reproduce failures.

---

## 3. Token Budget Exhaustion

**What it is:** Every conversation adds tokens to the context. For agents that maintain conversation history or retrieve large chunks of RAG content, the context window fills faster than expected — leading to truncation, degraded reasoning, or hard errors from the LLM provider.

The failure mode is insidious: the agent starts dropping the oldest parts of its context to make room, which often includes the system prompt instructions that govern its behaviour. It becomes a different agent mid-conversation.

**How to fix it:** Set hard token budgets per turn and per session. Implement context compression — summarising older turns rather than keeping them verbatim. Monitor p95 token usage per session in production so you see the cliff coming before agents start falling off it.

---

## 4. RAG Knowledge Staleness

**What it is:** Retrieval-Augmented Generation (RAG) gives agents access to a knowledge base. But knowledge bases go stale. If your vector store is not being re-indexed as your source data changes, the agent answers questions based on outdated information — with the same confidence as if the information were current.

A legal compliance agent trained on regulations from six months ago will answer questions about the current rules as if nothing has changed. A product support agent whose knowledge base has not been updated since a major release will confidently describe features that no longer work the way it says they do.

**How to fix it:** Schedule automated re-indexing on a cadence appropriate to how frequently your source data changes. For fast-moving domains, that may be daily. For stable domains, weekly. Add freshness metadata to your chunks so the retrieval layer can deprioritise stale content.

---

## 5. API Rate-Limit Cascades

**What it is:** Your agent calls external APIs. External APIs have rate limits. When usage spikes — a marketing campaign, a product launch, an unexpected viral moment — those rate limits are hit. If the agent has no circuit-breaker and no retry-with-backoff logic, a single rate-limit error cascades into a flood of failed retries that exhausts the limit further and brings the agent down entirely.

This is almost never caught in testing because test environments do not simulate production traffic volumes.

**How to fix it:** Implement retry logic with exponential backoff on every external API call. Set per-user and per-session call rate limits at the agent layer so a single misbehaving session cannot exhaust your quota. Add a circuit-breaker that puts the agent into a graceful degradation mode — limited functionality, not total failure — when a critical API is unavailable.

---

## The Common Thread

All five failure modes share one characteristic: they are invisible during development and emerge only under the conditions you cannot fully simulate before go-live. The fix is not to test harder in staging — it is to build observability and resilience into the agent itself so that when production does what production always does, your agent handles it without you needing to be paged at 2am.
